How I Think

Opinions I've earned
by getting things wrong.

01

Start with the problem, not the framework.

Reaching for a tool before understanding the problem is how you end up with a React app that should have been a static page, or a microservice that should have been a function. I try to ask "what does this actually need to do?" before I open a terminal.

02

Write backend code as if the next person is having a bad day.

Clear variable names, predictable function behaviour, errors that tell you what went wrong and where. Not because it's elegant — because at 2am when something breaks, you want to read code, not decode it.

03

UI is a product decision, not a decoration layer.

Every layout choice, every interaction, every piece of copy is a decision about how someone experiences your product. Treating frontend as "just making it look nice" is how you ship interfaces that work technically but fail the person using them.

04

Learning in public is faster than learning in private.

Writing about what you're learning — even badly — forces you to understand it well enough to explain it. The feedback loop is faster, the retention is better, and occasionally someone tells you you're wrong in a way that saves you weeks.

05

AI raises the floor on execution. The ceiling is now entirely about how well you think.

Anyone can generate working code in 2025. What you can't generate is the judgement to know what's worth building, the taste to know when something is wrong, and the clarity to explain a decision to someone who has to live with it. That's what I'm actually developing.

How I Use AI

Not a shortcut.
A way of thinking faster.

What I use it for

  • Compressing the distance between idea and working prototype
  • Exploring tradeoffs in architecture before committing to one
  • Reviewing my own code for edge cases I might have missed
  • Understanding unfamiliar codebases or concepts faster
  • Generating boilerplate so I can focus on the parts that actually require judgement

What I don't trust it for

  • Deciding what's worth building in the first place
  • Security-sensitive logic — I always read and understand that myself
  • The final call on architecture — it suggests, I decide
  • Anything where being wrong has a real cost
  • Replacing the thinking that makes the output actually good

"I use AI to compress the distance between idea and working code. I use my own judgement to decide if the idea was worth building in the first place."

01

Interfaces worth using

React Tailwind JS

The problem

Most interfaces are built for the developer's mental model, not the user's. The result is UIs that technically work but feel like work to use — confusing flows, inconsistent feedback, layouts that break on anything other than a 1440px screen.

My approach

Start with the user's goal, not the component library. Build responsive from the start, not as an afterthought. Treat every interaction — hover, focus, error state — as part of the product, not a detail to polish later.

What I'd build

A dashboard or data-heavy interface where the challenge isn't just displaying information, but making it scannable, actionable, and fast — even on a slow connection or a small screen.

02

Systems that hold up

Node.js Express MongoDB

The problem

Backend code that works in development has a habit of failing in production in ways nobody predicted. Unhandled edge cases, brittle database queries, APIs that return different shapes depending on the mood of the server.

My approach

Design APIs contract-first — know what they return before writing the logic. Handle errors explicitly, not optimistically. Keep the data layer separate from the business logic so neither becomes a liability when requirements change.

What I'd build

A REST API for a real-world domain — authentication, resource management, role-based access — with clean error handling, consistent response shapes, and a structure that doesn't collapse when a new feature gets added.

03

Data that means something

Python AI / ML SQL

The problem

Most applications collect data and do nothing useful with it. It sits in a database, gets queried for basic reports, and never informs a single product decision. The gap between "we have data" and "we understand our users" is enormous.

My approach

Currently learning the fundamentals — how models are trained, how to evaluate them honestly, and how to integrate ML outputs into a product without making the product dependent on the model being right 100% of the time.

What I'm exploring

Small, focused ML experiments — classification problems, data cleaning pipelines, simple predictive models — that build intuition for what's actually hard about making data useful in a production context.

Learning Log

What I'm figuring out,

Not a blog. No essays. Just short, honest notes on what I'm learning and how it's changing how I think. Updated manually — no algorithm, no schedule.

Apr 2025 AI / ML

The difference between a model that's accurate and one that's useful.

High accuracy on a test set doesn't mean a model is useful in production. This week I started understanding why — distribution shift, class imbalance, the gap between what a model optimises for and what the product actually needs. It's made me more skeptical of benchmark numbers.

Mar 2025 Backend

Why I stopped treating error handling as an afterthought.

I spent an afternoon debugging an Express API that was silently swallowing errors and returning 200s for failed operations. The fix was ten lines. The lesson was: explicit error handling isn't defensive programming, it's just honest programming.

Feb 2025 Frontend

CSS is not a styling language. It's a layout system with styling attached.

I used to fight CSS. Then I stopped thinking about it as "making things look right" and started thinking about it as "describing relationships between boxes." Once that clicked, Flexbox and Grid stopped feeling like magic and started feeling like logic.
